Mastering Bash Scripting

By Angix Black

Introduction to Bash Scripting

Bash, short for Bourne Again SHell, is a powerful Unix shell and command-line interpreter that extends the capabilities of the original Bourne Shell (sh). It’s the default shell on most Linux distributions and macOS, making it a critical tool for anyone working in a Unix-like environment. Bash scripting allows you to automate repetitive tasks, manage systems, and create powerful workflows by combining simple commands into reusable programs.

  • What is Bash?
    • A shell that interprets commands you type into a terminal.
    • A scripting language where you can write sequences of commands in a file (a script) to execute them as a single unit.
    • Built on top of the original Bourne Shell, Bash adds features like command history, job control, and advanced scripting capabilities.
  • Why Master Bash?
    • Automation: Turn time-consuming manual tasks—like file renaming or backups—into one-line scripts.
    • System Management: Manage users, monitor processes, or configure servers efficiently.
    • Development: Automate build processes, run tests, or deploy applications with precision.
    • Efficiency: Bash scripts are lightweight and fast, often requiring less overhead than compiled languages like C++ for system-level tasks.
    • Universality: Bash is ubiquitous on Unix systems, ensuring your skills are portable across platforms.

Here’s a simple Bash command to get your feet wet—type this into your terminal to see it in action:

echo "Hello, Bash!"

This command uses echo to print text to the screen. It’s the simplest way to output something in Bash and a building block for more complex scripts.

Getting Started

Writing your first Bash script is straightforward once you understand the basics. A script is just a text file containing commands you’d normally type into a terminal, plus a special line at the top to tell the system how to run it. Let’s walk through the process step-by-step.

1. Crafting Your First Script

  • Editor: Use a text editor like nano (simple and beginner-friendly), vim (powerful but with a learning curve), or even a graphical editor like VS Code.
  • Shebang: Start your script with #!/bin/bash. This line, called a shebang or hashbang, tells the system to use the Bash interpreter located at /bin/bash to execute the file.
  • Example: Here’s a basic script that prints a message and the current date:
#!/bin/bash
# Print a greeting and current date
echo "Hello, Bash scripting world!"
date

In this script, # starts a comment (ignored by Bash), echo outputs text, and date is a command that displays the current date and time. Save this as, say, my_script.sh.

2. Making It Executable

Unix systems require you to mark a script as executable before you can run it directly. Use the chmod (change mode) command to add execute permissions: 

chmod +x my_script.sh

Here, +x adds the "execute" permission to the file for the current user. Without this, you’d need to run it explicitly with bash my_script.sh.

3. Running It

Now, execute your script from the terminal by specifying its path: 

./my_script.sh

The ./ means "current directory," so Bash looks for my_script.sh where you’re currently located. You’ll see the greeting and date printed to the screen!

Core Bash Concepts

To write effective Bash scripts, you need to grasp its foundational elements. These concepts let you interact with the system, store data, and control script behavior.

1. Commands

Bash scripts are built around Unix commands—small programs like ls (list directory, though it’s dir on some systems), mkdir (make directory), or grep (search text). You can use them directly in scripts or capture their output. For example: 

echo "Current directory: $(pwd)"

Here, pwd (print working directory) outputs the current path, and $( ) captures that output to include it in the echo statement.

2. Variables

Variables store data for later use—no type declaration needed, just assign a value with =. Access them with a $ prefix. Here’s an example: 

#!/bin/bash
NAME="User"
echo "Hello, $NAME!"

NAME="User" sets the variable, and $NAME retrieves it. Note: no spaces around =, or Bash will misinterpret the command!

3. Comments

Comments explain your code, making it easier to maintain. Use # to start a comment—everything after it on the line is ignored by Bash: 

# This is ignored
echo "This runs"

Comments are crucial for documenting complex scripts or sharing code with others.

4. Input/Output

Bash scripts can interact with users via input and output. echo handles output, while read captures input into a variable: 

#!/bin/bash
echo "Enter your name:"
read NAME
echo "Hi, $NAME!"

When run, this script prompts you to type a name, stores it in NAME, and then greets you with it. It’s a simple way to make scripts interactive.

Control Flow

Control flow lets your script make decisions and repeat actions, turning simple command sequences into dynamic programs. Bash offers several constructs for this.

1. If Statements

Use if to execute code based on conditions. Conditions are tested with square brackets [ ], and the syntax ends with fi: 

#!/bin/bash
AGE=20
if [ $AGE -ge 18 ]; then
    echo "Adult"
else
    echo "Minor"
fi

-ge means "greater than or equal to." Spaces around [ and ] are mandatory—Bash treats [ as a command! This checks if AGE is 18 or older.

2. Case Statements

For multiple conditions, case is cleaner than nested ifs. It matches a value against patterns: 

#!/bin/bash
echo "Pick a color (red, blue, green):"
read COLOR
case $COLOR in
    "red") echo "Fiery!" ;;
    "blue") echo "Cool!" ;;
    "green") echo "Lush!" ;;
    *) echo "Unknown color" ;;
esac

;; ends each case, and *) is a catch-all for unmatched inputs. This is great for menus or option handling.

3. Loops

For Loop:

Repeat actions over a list or range with for. The {1..3} syntax creates a range: 

#!/bin/bash
for i in {1..3}; do
    echo "Loop $i"
done

This loops three times, printing "Loop 1," "Loop 2," and "Loop 3." The do and done keywords bracket the loop body.

While Loop:

Use while to loop as long as a condition is true: 

#!/bin/bash
COUNT=0
while [ $COUNT -lt 3 ]; do
    echo "Count: $COUNT"
    COUNT=$((COUNT + 1))
done

-lt means "less than." The $(( )) performs arithmetic to increment COUNT until it reaches 3.

4. Functions

Functions group code for reuse. Define them with a name and curly braces, then call them by name: 

#!/bin/bash
greet() {
    echo "Hello, $1!"
}
greet "World"

$1 is the first argument passed to the function. Here, it prints "Hello, World!" Functions help organize scripts and reduce repetition.

Operators & Syntax

Bash’s power comes from its operators and syntax, which let you manipulate data and control execution beyond basic commands. This includes comparison operators for numbers, strings, and files—key to writing conditional logic.

1. String Operations

Combine or manipulate strings with variables using curly braces ${} for clarity or when concatenating: 

TEXT="Hello"
echo "${TEXT}, Bash!"

${TEXT} ensures Bash reads the variable name correctly, especially useful if followed by text (e.g., ${TEXT}world vs. $TEXTworld). This outputs "Hello, Bash!"

2. Arithmetic

Perform math with $(( )), which evaluates expressions: 

#!/bin/bash
A=5
B=3
echo $((A + B))

This adds A and B, printing "8." Bash supports +, -, *, /, and % (modulus), but only integer arithmetic by default.

3. Command Substitution

Capture command output with $( ) or backticks `` (though $( ) is preferred for readability): 

NOW=$(date)
echo "It’s $NOW"

$(date) runs the date command and stores its output (e.g., "Fri Mar 28 2025...") in NOW. This is key for dynamic scripts.

4. Comparison Operators

Bash uses operators within [ ] or [[ ]] for conditionals (e.g., in if statements). These fall into three categories: numeric, string, and file tests. Note: Spaces around operators and brackets are mandatory!

Numeric Comparisons

Used for comparing integers:

  • -eq: Equal to (e.g., [ 5 -eq 5 ] is true).
  • -ne: Not equal to (e.g., [ 5 -ne 6 ] is true).
  • -gt: Greater than (e.g., [ 7 -gt 3 ] is true).
  • -ge: Greater than or equal to (e.g., [ 5 -ge 5 ] is true).
  • -lt: Less than (e.g., [ 2 -lt 9 ] is true).
  • -le: Less than or equal to (e.g., [ 4 -le 4 ] is true).
#!/bin/bash
NUM=10
if [ "$NUM" -ge 5 ]; then
    echo "$NUM is greater than or equal to 5"
fi

This checks if NUM is ≥ 5. Quote variables ("$NUM") to avoid errors if they’re unset.

String Comparisons

Used for comparing strings (use = and != in [ ], or == in [[ ]]):

  • =: Equal to (e.g., [ "abc" = "abc" ] is true).
  • !=: Not equal to (e.g., [ "abc" != "def" ] is true).
  • -z: Zero length (empty) (e.g., [ -z "" ] is true).
  • -n: Non-zero length (not empty) (e.g., [ -n "text" ] is true).
  • < and >: Lexicographical order (in [[ ]] only, e.g., [[ "apple" < "banana" ]] is true).
#!/bin/bash
STR="hello"
if [ "$STR" = "hello" ]; then
    echo "String matches!"
fi
if [ -n "$STR" ]; then
    echo "String is not empty!"
fi

This tests string equality and length. Use [[ ]] for safer string comparisons (e.g., handles spaces better).

File Test Operators

Check file properties:

  • -e: Exists (e.g., [ -e "/path" ]).
  • -f: Exists and is a regular file (e.g., [ -f "file.txt" ]).
  • -d: Exists and is a directory (e.g., [ -d "/home" ]).
  • -r: Readable (e.g., [ -r "file.txt" ]).
  • -w: Writable (e.g., [ -w "file.txt" ]).
  • -x: Executable (e.g., [ -x "script.sh" ]).
  • -s: Exists and has size > 0 (e.g., [ -s "file.txt" ]).
#!/bin/bash
FILE="test.txt"
if [ -f "$FILE" ]; then
    echo "$FILE is a regular file"
elif [ -d "$FILE" ]; then
    echo "$FILE is a directory"
else
    echo "$FILE doesn’t exist or is something else"
fi

This checks if test.txt is a file or directory—super useful for scripts handling files!

Debugging Bash Scripts

Bugs happen—Bash provides tools to find and fix them. Debugging ensures your scripts run reliably.

  • -x Flag: Run your script with tracing to see each command as it executes:
    • bash -x my_script.sh

    This outputs every line prefixed with +, showing variable values and command expansions—perfect for spotting where things go wrong.

  • set -e: Exit immediately if any command fails (returns a non-zero status):
    • #!/bin/bash
set -e
echo "This runs"
false # Script stops here
echo "This won’t run"

    false always fails, so set -e halts the script before the last echo. Use this to catch errors early.

Best Practices

Writing good Bash scripts means making them readable, reliable, and maintainable. Follow these tips:

  • Meaningful Names: Use descriptive variable names like FILE_COUNT instead of x.
  • Comments: Explain why code exists, not just what it does—e.g., “# Ensure backup is unique” vs. “# Set TIMESTAMP”.
  • Quote Variables: Use "$VAR" to handle spaces or empty values safely—unquoted variables can break scripts.
  • Test Safely: Run scripts in a sandbox or with echo statements to preview actions before modifying files.
#!/bin/bash
# Backup files with timestamp to avoid overwrites
TIMESTAMP=$(date +%Y%m%d)
cp "$HOME/docs" "$HOME/docs_backup_$TIMESTAMP"

This script quotes "$HOME/docs" to handle spaces in paths and uses a timestamp for unique backups—simple but robust.

Exploring the Bash Manual

The Bash Manual is the definitive resource for Bash’s features and syntax. It covers everything from basic commands to advanced topics like arrays, parameter expansion, and signal handling—perfect for deepening your understanding.

For example, the manual explains arrays, which let you store multiple values in one variable. Here’s how to use them: 

#!/bin/bash
# Declare an array of fruits
FRUITS=("apple" "banana" "orange")
# Access all elements
echo "All fruits: ${FRUITS[@]}"
# Access one element
echo "First fruit: ${FRUITS[0]}"

${FRUITS[@]} prints all array elements, while ${FRUITS[0]} gets the first (arrays are zero-indexed). The manual details this and more, like how to loop over arrays or use associative arrays (key-value pairs) in Bash 4+.

Automating Real Tasks

Bash shines when automating real-world tasks. Think about daily chores like backing up files, cleaning up old logs, or checking system health. Here’s a practical example—a script to back up a directory with a timestamp and log the action: 

#!/bin/bash
# Backup directory with date and log it
SOURCE="$HOME/documents"
DEST="$HOME/backups"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
LOG="$HOME/backup_log.txt"

# Create backups directory if it doesn’t exist
mkdir -p "$DEST"
# Copy files
cp -r "$SOURCE" "$DEST/documents_$TIMESTAMP"
# Log the action
echo "Backup created at $TIMESTAMP" >> "$LOG"

This script uses -p with mkdir to avoid errors if the directory exists, -r with cp to copy recursively, and >> to append to the log. Run it with cron (e.g., 0 2 * * * /path/to/script.sh) for daily 2 AM backups!

Advanced Tools: awk and sed

Bash pairs beautifully with tools like awk (text processing) and sed (stream editing) to handle complex data manipulation. They’re command-line powerhouses you’ll find on most Unix systems.

1. awk

awk processes text line-by-line, splitting it into fields. It’s great for extracting data from structured output: 

#!/bin/bash
# Print usernames from /etc/passwd
awk -F: '{print $1}' /etc/passwd

-F: sets the field separator to a colon, and $1 grabs the first field (username) from each line of /etc/passwd. This lists all system users.

2. sed

sed edits text streams—perfect for search-and-replace or line modifications: 

#!/bin/bash
# Replace "hello" with "hi" in a file
sed 's/hello/hi/g' input.txt > output.txt

s/hello/hi/g substitutes "hello" with "hi" globally (g) on each line, redirecting the result to output.txt. Combine it with pipes for real-time edits!

Signal Handling

Bash can trap signals (e.g., Ctrl+C) to handle interruptions gracefully. Use the trap command to define actions for specific signals: 

#!/bin/bash
# Trap Ctrl+C (SIGINT) and clean up
cleanup() {
    echo "Script interrupted! Cleaning up..."
    rm -f /tmp/tempfile
    exit 1
}
trap cleanup INT

echo "Running... (Press Ctrl+C to interrupt)"
touch /tmp/tempfile
sleep 10
echo "Done!"

trap cleanup INT runs the cleanup function when SIGINT (interrupt) is received. This script creates a temp file, waits, and cleans up if interrupted—useful for ensuring resources are freed.

Process Management

Bash lets you manage processes with commands like & (background), jobs, fg, bg, and kill. Here’s a script to run a task in the background and manage it: 

#!/bin/bash
# Run a long task in the background
echo "Starting background task..."
sleep 30 & # Background process
PID=$!    # Capture process ID

echo "Background PID: $PID"
sleep 2
echo "Killing background task..."
kill $PID
wait $PID 2>/dev/null # Wait for it to finish, suppress error
echo "Task stopped."

& runs sleep in the background, $! gets its PID, and kill stops it. wait ensures the script waits for completion—great for managing parallel tasks.

Pipelines

Pipelines (|) connect commands, passing output from one as input to the next. They’re a cornerstone of Unix philosophy. Here’s an example to count unique users in a log: 

#!/bin/bash
# Count unique users from a log file
LOG="/var/log/auth.log"
echo "Unique users in $LOG:"
cat "$LOG" | grep "session opened" | awk '{print $11}' | sort | uniq | wc -l

This pipeline: - cat reads the log. - grep filters session openings. - awk extracts the username (field 11). - sort | uniq removes duplicates. - wc -l counts the lines (unique users). Pipelines chain commands efficiently, avoiding temporary files.

Parameter Expansion

Parameter expansion lets you manipulate variables with advanced syntax, like setting defaults or modifying strings. It uses ${} with special operators: 

#!/bin/bash
# Set a default value if variable is unset
NAME=${1:-"Guest"}  # Use first argument or "Guest"
echo "Hello, $NAME!"

# Remove prefix or suffix
FILE="report.txt"
echo "Base name: ${FILE%.txt}"  # Removes .txt

# Length of a string
TEXT="Bash"
echo "Length: ${#TEXT}"  # Prints 4

- ${VAR:-default} uses "default" if VAR is unset or null. - ${VAR%.suffix} removes the shortest match of ".suffix" from the end. - ${#VAR} gives the string length. These are powerful for dynamic scripts!

Error Handling

Robust scripts handle errors gracefully. Check exit codes ($?) and provide feedback instead of failing silently: 

#!/bin/bash
# Check if a file exists and copy it
SOURCE="important.txt"
DEST="backup.txt"

if [ ! -f "$SOURCE" ]; then
    echo "Error: $SOURCE not found!" >&2
    exit 1
fi

cp "$SOURCE" "$DEST"
if [ $? -ne 0 ]; then
    echo "Error: Copy failed!" >&2
    exit 1
else
    echo "Copy succeeded!"
fi

- [ ! -f "$SOURCE" ] tests if the file doesn’t exist. - $? holds the last command’s exit code (0 = success, non-zero = failure). - &>2 sends errors to stderr. This ensures users know what went wrong.

Project: Log Analyzer

Let’s tie it all together with a simple log analyzer that counts failed login attempts from /var/log/auth.log: 

#!/bin/bash
# Log analyzer for failed login attempts
LOG="/var/log/auth.log"
OUTPUT="failed_logins_$(date +%Y%m%d).txt"

# Check if log exists
if [ ! -f "$LOG" ]; then
    echo "Error: $LOG not found!" >&2
    exit 1
fi

# Trap Ctrl+C for cleanup
cleanup() {
    echo "Interrupted! Cleaning up..." >&2
    rm -f "$OUTPUT"
    exit 1
}
trap cleanup INT

echo "Analyzing failed login attempts..."
grep "Failed password" "$LOG" > "$OUTPUT"
COUNT=$(wc -l < "$OUTPUT")

if [ $COUNT -eq 0 ]; then
    echo "No failed attempts found."
    rm -f "$OUTPUT"
else
    echo "Found $COUNT failed login attempts. Details in $OUTPUT."
fi

This script: - Checks for the log file (error handling). - Uses a trap for cleanup (signal handling). - Filters failed logins with grep (pipelines). - Saves results with a timestamp (automation). - Reports the count or cleans up if none found (control flow). Run it with sudo if needed for log access—a real-world tool!
